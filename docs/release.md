# Release runbook

## Cutting a release

1. Update `CHANGELOG.md` with the new version entry.
2. Tag the commit in the public repo: `git tag v0.x.y && git push --tags`
3. The `release.yml` workflow fires, builds cross-platform archives, publishes the GitHub Release, and commits a fresh `bcdock.rb` formula to `bcdock/homebrew-tap`.

Verify the release landed: check the [Releases page](https://github.com/bcdock/cli/releases) and the [tap commit history](https://github.com/bcdock/homebrew-tap/commits/main).

## Homebrew tap setup

One-time setup required before the first tagged release. Both steps are Alex actions.

**Step 1 - Create the tap repo.** Create `github.com/bcdock/homebrew-tap` as an empty repo (MIT license, no other content). Goreleaser commits the formula on first tag push; the tap repo must exist beforehand or goreleaser exits with "repo not found."

Disable branch protection on the tap repo. It is a generated artifact - goreleaser pushes directly to the default branch. PR-review protection breaks the push.

**Step 2 - Add the `HOMEBREW_TAP_TOKEN` secret.** Create a fine-grained PAT with `Contents: Read+Write` scoped to `bcdock/homebrew-tap` only. No other repo permissions; no org permissions. Set a minimum lifetime of 90 days and note the expiry - rotate before it lapses or goreleaser push fails silently on the next release.

Add the PAT as a repo secret on `bcdock/cli`: Settings - Secrets and variables - Actions - New repository secret - name it `HOMEBREW_TAP_TOKEN`.

**Verification.** After the first real `v*` tag:

```bash
brew tap bcdock/tap
brew install bcdock
bcdock version
```

The installed version string should match the tag.

## npm trusted publishing

`@bcdock/cli` is published to npmjs.com via OIDC trusted publishing - no long-lived `NPM_TOKEN` secret required.

**How it works.** The `npm-publish` job in `release.yml` has `id-token: write` in its `permissions:` block. GitHub mints a short-lived OIDC token scoped to this workflow run; npm's registry validates it against the trusted-publisher entry configured at npmjs.com (package `@bcdock/cli`, repo `bcdock/cli`, workflow `release.yml`). No secret rotation, no 2FA dance.

**Trusted publisher config** (one-time, already done). npmjs.com - package `@bcdock/cli` - Publishing - Trusted publishers - GitHub Actions. Configured by Alex on 2026-05-19.

**Post-migration cleanup** (Alex action, after first successful OIDC-based publish). Delete the `NPM_TOKEN` secret from `bcdock/cli` Settings - Secrets and variables - Actions. The workflow no longer references it.

## Notes

- The formula is entirely generated by goreleaser. Never hand-edit `bcdock/homebrew-tap/bcdock.rb` - the next release overwrites it.
- If you delete and re-push the same tag (e.g., to fix a botched release), goreleaser re-commits the formula. The tap history shows two commits for the same version; Homebrew users are unaffected.
- macOS Gatekeeper: Homebrew strips the quarantine attribute on install, so unsigned binaries work. Manual archive downloads may show a Gatekeeper warning on first run - this is a known limitation (code signing is out of scope until stable release cadence).

## Release the CLI (one-click flow)

The release-cli flow lives in two `workflow_dispatch` workflows in the monorepo `.github/workflows/`:

1. **`Release CLI`** (`release-cli.yml`) - main-only. Input: `version` (e.g. `0.1.0-rc4`).
   Steps automatically: validate, generate cli-scoped changelog from the last stable `cli/v*` tag (cumulative across rcs - falls back to the last prerelease if no stable exists yet), create + push the `cli/v<version>` annotated tag (fires `mirror-cli.yml`), create the monorepo GitHub Release, wait for goreleaser on the public side, then overwrite the public release body with the same notes (install header + footer-links preserved). No commit author names in either release body.
2. **`Delete CLI Release`** (`delete-cli-release.yml`) - main-only. Inputs: `version` + `confirm` (must match). Deletes the release+tag from both repos. npm + Homebrew cleanup is NOT automated; the workflow summary prints the manual `npm deprecate` command.

Typical flow for a new rc:
- Actions -> **Release CLI** -> Run workflow -> version=`0.1.0-rc4` -> click Run
- ~7-10 minutes later, both releases are live with cli-scoped notes
- If something's wrong: Actions -> **Delete CLI Release** -> version=`0.1.0-rc4`, confirm=`0.1.0-rc4` -> Run

The orchestrator refuses to run if the tag already exists on either repo. Bump version or delete first.
